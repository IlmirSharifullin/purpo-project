# Архитектура системы

## 1. Выбор паттерна мультиагентной системы

### Сравнение архитектурных подходов

| Паттерн | Описание | Когда оптимален |
|---|---|---|
| **Supervisor (выбрано)** | Центральный агент-маршрутизатор, делегирующий специалистам | Фиксированные домены, явная маршрутизация, нужна синтез ответов от нескольких агентов |
| Flat ReAct (один агент, все инструменты) | Один агент со всеми инструментами | Менее 5 инструментов, специализация не нужна |
| Hierarchical (supervisor of supervisors) | Супервайзеры второго уровня | Более 20 агентов, суб-домены со своими координаторами |
| Swarm (peer-to-peer handoffs) | Агенты передают управление друг другу без центра | Исследовательские задачи, непредсказуемый порядок |
| Plan-and-Execute | Планировщик + исполнитель | Долгосрочные задачи, требующие предварительного плана |

### Обоснование выбора Supervisor

Четыре чётко разграниченных специалиста с непересекающимися наборами инструментов — классический кандидат для Supervisor:
- Каждый агент имеет строго определённую область: рецепты / КБЖУ / покупки / готовка
- Маршрутизация предсказуема: тип запроса однозначно определяет агента
- Супервайзер синтезирует ответы нескольких агентов в единый ответ

**Основная проблема (выявлена по output.json):** модель qwen2.5:3b иногда отвечает сама вместо делегирования. Это проблема системного промпта, а не архитектуры.

## 2. Выбор фреймворка: LangGraph

### Сравнение с альтернативами

| Фреймворк | Pros | Cons |
|---|---|---|
| **LangGraph (выбрано)** | Граф состояния, нативная поддержка checkpointing, production-grade | Сложнее для простых кейсов |
| AutoGen | Мощные multi-agent conversations | Менее гибкий граф, привязка к OpenAI |
| CrewAI | Удобный DSL | Ограниченная кастомизация потока |
| LangChain LCEL | Простые цепочки | Нет нативного состояния между агентами |
| Semantic Kernel | .NET-first | Не Python-native |

**Почему LangGraph:**
- `create_supervisor` из `langgraph_supervisor` — готовый паттерн в 5 строк
- `SqliteSaver` / `InMemorySaver` — персистентная память без усилий
- Полный граф выполнения в `output_mode="full_history"` → удобен для evals
- Активно поддерживается (LangChain Inc.)

## 3. Выбор LLM: qwen2.5:3b через Ollama

### Сравнение открытых моделей

| Модель | Размер | Скорость на CPU | Качество RU | Требования RAM |
|---|---|---|---|---|
| **qwen2.5:3b (выбрано)** | 3B | ~15 tok/s CPU | Хорошее | 4 ГБ |
| llama3.2:3b | 3B | ~15 tok/s CPU | Хуже RU | 4 ГБ |
| mistral:7b | 7B | ~7 tok/s CPU | Хорошее | 8 ГБ |
| qwen2.5:7b | 7B | ~7 tok/s CPU | Отличное | 8 ГБ |
| llama3.1:8b | 8B | ~6 tok/s CPU | Среднее | 10 ГБ |

**Почему qwen2.5:3b:** лучший баланс качества русского языка и скорости инференса при ограниченных ресурсах. Ollama поддерживает GPU-ускорение при наличии.

## 4. Стратегия изоляции

### Сравнение всех вариантов

| Вариант | Pros | Cons | Вердикт |
|---|---|---|---|
| **venv (текущее)** | Нулевой overhead, мгновенный старт | Ollama не версионирован, нет воспроизводимости на других машинах | Dev only |
| **Docker (только app)** | Воспроизводимый Python env | Всё ещё требует Ollama на хосте | Частичное решение |
| **Docker Compose app+Ollama (выбрано)** | Полная воспроизводимость, версионирование Ollama, GPU passthrough `--gpus all`, изоляция портов | ~2 ГБ образ Ollama, требует Docker Desktop на macOS | **Prod & CI** |
| VM (UTM/Lima) | Полная изоляция ОС | Латентность +100–200мс для LLM через сеть, массивный overhead | Избыточно |
| Firecracker / gVisor | Production security | Экстремальная сложность, нет пользы для локальной разработки | Избыточно |
| conda env | ML-зависимости | Не нативен для uv, избыточно | Не нужен |

**Итоговая стратегия:**
- **Разработка:** чистый venv через `uv`
- **CI/Деплой:** Docker Compose (`docker-compose up`)
- GPU passthrough раскомментировать в `docker-compose.yml` при наличии NVIDIA GPU

## 6. Замечания по масштабированию

При росте системы рекомендуется:
1. **> 4 агентов:** перейти на иерархический Supervisor (supervisor of supervisors)
2. **> 1000 рецептов:** добавить RAG на ChromaDB для семантического поиска
3. **Multi-user concurrent:** заменить SqliteSaver на `AsyncPostgresStore`
4. **Production LLM:** заменить Ollama на vLLM или Triton Inference Server
