# Техническое задание (ТЗ)
## Кулинарный ассистент на основе мультиагентной системы

**Версия:** 1.0  
**Дата:** 2026-04-14

---

## 1. Назначение системы

Интеллектуальный кулинарный помощник, который помогает пользователю:
- подбирать рецепты по имеющимся продуктам
- рассчитывать пищевую ценность (КБЖУ)
- составлять списки покупок и оценивать бюджет
- получать советы по процессу приготовления

---

## 2. Актёры

| Актёр | Описание |
|---|---|
| Пользователь | Человек, общающийся с системой через CLI или API |
| LLM (Ollama) | Локальная языковая модель (qwen2.5:3b) |
| Супервайзер | Агент-маршрутизатор, делегирующий запросы специалистам |
| Recipe Finder | Агент поиска рецептов |
| Nutritionist | Агент-диетолог |
| Grocery List | Агент составления списков покупок |
| Cooking Coach | Агент-наставник по готовке |

---

## 3. Функциональные требования

### FR-01: Поиск рецептов
- Система принимает список продуктов (через запятую) и возвращает до 3 рецептов
- Система возвращает пошаговый рецепт по названию блюда
- Обязательный вызов инструмента (не ответ из памяти модели)

### FR-02: Расчёт КБЖУ
- Система рассчитывает калории, белки, жиры, углеводы для продукта/блюда по граммам
- Система рекомендует суточный рацион под цели: похудение / набор массы / поддержание
- Точность: значения берутся из БД, не генерируются LLM

### FR-03: Список покупок
- Система формирует категоризированный список продуктов на неделю
- Система оценивает бюджет с разбивкой по категориям (в рублях)

### FR-04: Кулинарный наставник
- Система находит замену ингредиенту с пропорциями
- Система сообщает время и советы по приготовлению (метод: варка/жарка/запекание/пароварка)

### FR-05: Мультиагентная обработка
- Система корректно обрабатывает запросы, требующие нескольких агентов последовательно
- Каждый под-вопрос делегируется соответствующему специалисту

### FR-06: Память сессии
- Система сохраняет историю диалога между сообщениями в рамках сессии (thread_id)
- Состояние сессии персистентно (SqliteSaver) — выживает перезапуск процесса

### FR-07: Наблюдаемость
- Система экспортирует метрики в формате Prometheus (порт 8000)
- Система ведёт структурированные логи (JSON или консольный формат)

---

## 4. Нефункциональные требования

| Требование | Значение |
|---|---|
| Язык общения | Русский |
| LLM | Открытый исходный код, локальный инференс (Ollama) |
| Модель | qwen2.5:3b (минимум), совместимо с любой Ollama-моделью |
| Латентность | < 60с на запрос (при локальном CPU-инференсе) |
| Персистентность | SQLite, без внешних сервисов |
| Изоляция | Docker Compose (app + Ollama sidecar) |
| Конфигурация | Через `.env` файл |
| Воспроизводимость | `uv.lock` + Dockerfile |

---

## 5. Ограничения и допущения

- База рецептов и КБЖУ — статические справочники в коде (не внешняя БД)
- Отсутствует веб-интерфейс — только CLI и Python API
- Отсутствует авторизация пользователей
- InMemoryStore не сохраняет пользовательские профили после перезапуска процесса
- Точность ответов зависит от качества модели qwen2.5:3b

---

## 6. Структура системы

```
Пользователь
    │
    ▼
Супервайзер (LangGraph StateGraph)
    │
    ├──► Recipe Finder   [find_recipe_by_ingredients, get_recipe_details]
    ├──► Nutritionist    [calculate_nutrition, adjust_diet_for_goal]
    ├──► Grocery List    [create_weekly_grocery_list, estimate_budget]
    └──► Cooking Coach   [find_ingredient_substitute, get_cooking_time]
```

---

## 7. Критерии приёмки

- [ ] Все 4 агента корректно маршрутизируются супервайзером
- [ ] Инструменты вызываются при каждом соответствующем запросе
- [ ] Мультиагентные запросы обрабатываются полностью (обе части отвечены)
- [ ] Eval suite: pass_rate ≥ 80% на встроенных 9 тест-кейсах
- [ ] Метрики доступны на `http://localhost:8000/metrics`
- [ ] История диалога сохраняется между вызовами (один thread_id)
- [ ] Система запускается командой `docker-compose up`
