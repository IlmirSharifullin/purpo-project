# Система памяти

## 1. Анализ вариантов

### Сравнительная таблица

| Вариант | Плюсы | Минусы | Итог |
|---|---|---|---|
| **ConversationBufferMemory** | Простой drop-in | Устарел в LangChain v1+, не интегрирован с LangGraph state, растёт без ограничений | ❌ |
| **ConversationSummaryMemory** | Сжатие контекста | Та же проблема с LangGraph, требует лишнего LLM-вызова | ❌ |
| **JSON file-based** | Zero deps, читаемый | Race conditions при конкурентном доступе, нет транзакций, нет запросов | ❌ |
| **Redis** | Быстрый, TTL, pub/sub | Требует запущенного сервиса, добавляет сложность ops | ❌ |
| **ChromaDB RAG** | Семантический поиск, масштабируется | ~200 МБ зависимость, нужна embedding-модель, overhead для базы < 1000 рецептов | Отложен |
| **LlamaIndex + RAG** | Полноценный RAG-пайплайн | Те же требования к embeddings, дублирует LangChain функциональность | Отложен |
| **SQLite Checkpointer** (выбрано) | Нативен LangGraph, zero extra services, thread-safe, персистентен | Single-writer bottleneck | ✅ short-term |
| **InMemoryStore** (выбрано) | Нативен LangGraph, zero deps, K/V структура | Теряется при перезапуске процесса | ✅ long-term (dev) |

## 2. Выбранная архитектура: гибридная двухуровневая

```
┌─────────────────────────────────────────────┐
│  SHORT-TERM MEMORY (диалог)                  │
│  SqliteSaver                                  │
│  ─ Ключ: thread_id                           │
│  ─ Хранит: все сообщения + вызовы инструментов│
│  ─ Персистентен между перезапусками           │
│  ─ Файл: data/checkpoints.db                 │
└─────────────────────────────────────────────┘
                    +
┌─────────────────────────────────────────────┐
│  LONG-TERM MEMORY (профиль пользователя)     │
│  InMemoryStore                               │
│  ─ Ключ: ("users", user_id) / "profile"     │
│  ─ Хранит: диетические цели, ограничения     │
│  ─ Теряется при перезапуске процесса         │
└─────────────────────────────────────────────┘
```

**Интеграция в LangGraph:**
```python
checkpointer = build_checkpointer()   # SqliteSaver
store = get_store()                   # InMemoryStore
app = workflow.compile(checkpointer=checkpointer, store=store)

# Вызов с контекстом конкретной сессии:
result = app.invoke(
    {"messages": [...]},
    config={"configurable": {"thread_id": "session-abc"}},
)
```

## 3. Недостатки выбранного подхода

1. **InMemoryStore не персистентен** — профили пользователей теряются при перезапуске.  
   *Решение:* заменить на `langgraph-checkpoint-postgres` `AsyncPostgresStore` (одна строка кода).

2. **SQLite single-writer** — конкурентные сессии сериализуются на записи.  
   *Решение:* при необходимости конкурентного доступа — PostgreSQL checkpointer.

3. **Нет семантического поиска** — система не может найти рецепт по описанию («что-то лёгкое и быстрое»).  
   *Решение:* см. Раздел 4 (RAG).

4. **Нет TTL** — старые сессии накапливаются в SQLite бесконечно.  
   *Решение:* периодическая очистка `DELETE FROM checkpoints WHERE timestamp < ...`.

## 4. Путь обновления: RAG

RAG (Retrieval-Augmented Generation) оправдан, когда:
- База рецептов вырастает до > 1000 позиций
- Нужен семантический поиск («блюда без лактозы», «быстрый завтрак»)
- Пользователь описывает блюдо, не называя продукты

**Рекомендуемый стек для RAG-upgrade:**
```
ChromaDB (локальный векторный store)
    + sentence-transformers (локальные embeddings, например BAAI/bge-m3)
    + LangChain VectorStoreRetriever
    → Заменяет/дополняет find_recipe_by_ingredients
```

**Почему не сейчас:**
- 20 рецептов в статической БД не оправдывают +200 МБ зависимостей
- Добавление RAG не требует изменений в агентном слое — только замена инструмента

**Сравнение RAG-фреймворков:**

| Фреймворк | Описание | Подходит |
|---|---|---|
| LangChain VectorStore | Встроен, минимум кода | ✅ при текущем стеке |
| LlamaIndex | Специализирован на RAG, больше возможностей | ✅ при сложном RAG |
| Haystack | Production-grade, но тяжелее | Для enterprise |
| RAGAS | Только evaluation RAG | Для оценки качества |

## 5. Управление контекстом (trim_messages_to_window)

Модель qwen2.5:3b имеет ограниченный контекст (4096 токенов). При длинных сессиях используется `trim_messages_to_window()`:
- Сохраняет первое системное сообщение
- Оставляет последние N сообщений (по умолчанию 20)
- Предотвращает переполнение контекста
